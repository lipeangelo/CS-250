## Overview
This project was developed as part of **CS 250: Software Development Lifecycle** at Southern New Hampshire University.  
The goal was to design and implement a travel application using **Scrum-Agile methodology**, while taking on multiple roles within a Scrum team.

The application showcases a **Top 5 Travel Destinations** feature using both:
- A **List View Interface**
- A **Slide Show Interface (Updated Requirements)**


## Features
- Top 5 travel destinations display
- Images with descriptions
- Clickable links to travel packages
- Slide show navigation (Next / Previous)
- UI customization and styling
- Functional validation through test cases


## Agile & Scrum Implementation
Throughout this project, I applied key Agile practices:

- **User Stories** → Defined features based on user needs  
- **Sprint Iterations** → Built features incrementally  
- **Test Cases** → Validated functionality and requirements  
- **Adaptability** → Updated UI from list → slideshow  
- **Retrospective Analysis** → Evaluated process and improvements

## Portfolio Reflection

### How do I interpret user needs and implement them into a program? How does creating “user stories” help with this?

Interpreting user needs starts with understanding what the user wants to accomplish and why it matters. Instead of focusing only on technical details, I translate those needs into user stories, which describe the feature from the user’s perspective. This helps break down complex requirements into smaller, clear, and actionable tasks.

User stories also include acceptance criteria, which define what success looks like for each feature. In this project, for example, the “Top Five Destinations” feature required images, descriptions, and links. These details ensured that the final implementation matched both functional requirements and user expectations. Overall, user stories help keep development focused on delivering real value rather than just writing code.

### How do I approach developing programs? What Agile processes do I hope to incorporate into my future development work?

I approach developing programs using an iterative and incremental process. Instead of building everything at once, I develop features step-by-step, test them, and improve them based on feedback. This approach reduces errors and makes it easier to adapt when requirements change.

During this course, I applied Agile practices such as breaking work into user stories, testing functionality continuously, and refining features over time. For example, I adapted the application from a list-based interface to a slideshow format without restarting the project, which shows the flexibility of Agile development.

In the future, I plan to continue using:

- User stories and backlog management to organize work
- Sprint-based development to deliver features incrementally
- Continuous testing and feedback loops to improve quality
- Retrospectives to reflect and improve my workflow

These practices will help me build more efficient, flexible, and user-focused software.

### What does it mean to be a good team member in software development?

Being a good team member in software development means contributing to both the technical work and the team environment. It involves clear communication, collaboration, accountability, and a willingness to adapt.

Throughout this project, I worked in multiple roles, which showed me how each team member contributes to the overall success of a project. A good team member:

- Communicates progress and challenges clearly
- Collaborates with others to solve problems
- Takes responsibility for their work
- Is open to feedback and continuous improvement

In Agile environments, teamwork is especially important because progress depends on constant communication and shared responsibility. By supporting the team and staying aligned with project goals, a developer can help deliver high-quality software more efficiently.
